# QML Deduplication Report (Stage 4)

Behavior-affecting refactors are **reported, not applied** (they change call sites
and cannot be render-verified in-session). Apply incrementally and build/render-test
each before trusting. The component library is already mature (24 extracted
components), so opportunities are modest and low-urgency.

## High-confidence candidates

### 1. `ConfirmDialog.qml` / `RenameTagDialog.qml` → use `ThemedDialog`
Both declare raw dialog scaffolding while `ThemedDialog.qml` exists and is used by
other dialogs. Consolidating gives consistent theming, backdrop, and button
styling for free, and removes duplicated header/footer markup.
- Risk: low. Verify the two dialogs' custom buttons map onto ThemedDialog's slots.
- Payoff: removes ~2 copies of dialog chrome; single source for dialog look.

### 2. Inline `radius: Platform.radius` rounded panels (7 files)
`Main.qml`, `HelpPage`, `NewsPage`, `ReviewPage`, `SettingsPage`, `StatsPage`,
`TagsPage` each hand-roll a `Rectangle { radius; border; color }` "card" surface.
`ChartCard.qml` already encapsulates this pattern.
- Candidate: extract a minimal `Card.qml` (or generalize `ChartCard`) and replace
  the seven inline panels.
- Risk: medium — each panel has slightly different padding/content; confirm each
  maps to the shared component's properties before replacing.
- Payoff: one surface primitive; consistent radius/border/elevation.

## Not recommended
- The 24 existing components (`ActionButton`, `IconBtn`, `TagChip`, `Stepper`,
  `ToggleSwitch`, etc.) are already well-factored; no merges warranted.
- Page-level layouts differ enough that a shared "page scaffold" would add
  indirection without real reuse.

## Method
Generated by structural grep, not semantic proof. Two components that look similar
may differ in one binding; each candidate above must be diffed and render-tested
per §5 (robustness) before the call sites are changed.
